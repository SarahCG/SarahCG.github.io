Asking Good Questions

8/14/15

I have had countless questions during phase 0 of Dev Bootcamp. Most times the answer was a simple google search away, but when it comes to something challenge specific, I did push out of my shy little bubble and post a question on our Google + community.

Here it is in all of it's glory -
If anyone has completed the variable scope challenge, do you have any suggestions for old code challenges to refactor?ï»¿

Oof. That's not a great question, Sarah. Kudos for being brave enough to ask, but let's see what we can do to improve it.

There are a few things about this question that we could improve.

  • Be specific

    • What answer are you looking for?
    Part of the challenge was to find and refactor older code using our newly developed understanding of class and variable scope. I had trouble finding any older code to work on, because up to that point our code challenges were pretty simple and our variables were mostly local.
    A better question would have given a more specific understanding of why I was struggling to find an old code challenge that would be appropriate to refactor. As it is, the question is pretty vague.

  • Be clear and concise

    • This question is pretty wishy washy and open ended. Even if people had examples of old code they chose to refactor, it may not be helpful seeing that everyone codes differently. It would be helpful to have given an example of what I tried and why it was not producing the results I wanted.

Refactored Question:

Variable Scope Challenge -
Having trouble finding previous code challenges to refactor since we've been working with local variables up to this point.
Can anyone suggest an old challenge they were able to refactor?

This is much more descriptive, and clearer in what it is actually asking. Admittedly this is a skill that I need to work on. The only bad questions are the ones that go unasked.
